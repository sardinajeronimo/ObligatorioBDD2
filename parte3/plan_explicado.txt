##Consulta
EXPLAIN PLAN FOR
SELECT 
    a.nombre                            AS agente,
    c.nombre                            AS comunidad,
    COUNT(DISTINCT p.id_contenido)      AS total_publicaciones,
    COUNT(v.id_voto)                    AS votos_positivos
FROM AGENTE a
    JOIN CONTENIDO ct   ON ct.id_agente    = a.id_agente
    JOIN PUBLICACION p  ON p.id_contenido  = ct.id_contenido
    JOIN COMUNIDAD c    ON c.id_comunidad  = p.id_comunidad
    LEFT JOIN VOTO v    ON v.id_publicacion = p.id_contenido
                       AND v.tipo = 'positivo'
WHERE ct.fecha_hora_creacion >= SYSDATE - 30
GROUP BY a.nombre, c.nombre
ORDER BY total_publicaciones DESC;
 
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

##Analisis

1. Operaciones identificadas**
Las operaciones del plan: HASH JOIN, NESTED LOOPS, TABLE ACCESS FULL, INDEX RANGE SCAN, HASH GROUP BY, SORT ORDER BY.

2. Relación con algoritmos vistos en clase**
- NESTED LOOPS → algoritmo nested loop join: por cada fila de una tabla busca en la otra usando índice
- HASH JOIN → algoritmo hash join: construye tabla hash de un lado, prueba el otro
- TABLE ACCESS FULL → full table scan: lee toda la tabla sin índice
- INDEX RANGE SCAN → usa el índice para leer solo las filas necesarias

3. Eficiencia**
- Costo total: 9 (muy bajo)
- Tiempo estimado: 1 segundo
- Oracle eligió nested loops donde hay índices disponibles
- Full scan en COMUNIDAD y AGENTE porque tienen pocas filas (4 y 6), es más barato que usar índice

4. Mejoras propuestas**
- Agregar índice en `CONTENIDO(fecha_hora_creacion)` para que el filtro de los 30 días use range scan en vez de filter
- Con más datos el optimizer podría cambiar de nested loops a hash join en algunos nodos 

### Relación con algoritmos vistos en clase

El plan utiliza dos algoritmos de join principales:

**Nested Loop Join** (líneas 7, 8, 13, 14): Oracle itera sobre cada fila de la tabla exterior y busca las filas correspondientes en la tabla interior usando un índice. Se usa cuando una de las tablas es pequeña o existe un índice eficiente. En este caso se aplica entre COMUNIDAD y PUBLICACION (usando IX_PUB_COMUNIDAD) y entre AGENTE y CONTENIDO (usando IX_CONTENIDO_AGENTE).

**Hash Join** (línea 6): Oracle construye una tabla hash en memoria con el resultado de un lado del join y luego prueba cada fila del otro lado contra esa tabla. Se usa cuando los conjuntos son medianos y no hay un índice que cubra el join. Aquí une el resultado de PUBLICACION+COMUNIDAD con el resultado de AGENTE+CONTENIDO.

**Hash Group By** (líneas 2, 4): Agrupa los resultados usando una tabla hash en memoria. Aparece dos veces por el COUNT(DISTINCT), que requiere una pre-agregación interna antes de la agregación final.

### Eficiencia

La consulta tiene un costo estimado de 9 con tiempo de ejecución de 1 segundo. Oracle tomó decisiones eficientes dado el volumen de datos actual:

- COMUNIDAD y AGENTE se leen completas (TABLE ACCESS FULL) porque tienen 4 y 6 filas respectivamente. Con tablas tan pequeñas, un full scan es más barato que acceder a un índice.
- Los índices IX_PUB_COMUNIDAD, IX_CONTENIDO_AGENTE e IX_VOTO_PUBLICACION se usan correctamente en los nested loops, evitando scans completos en las tablas más grandes.
- El LEFT JOIN a VOTO se implementa como NESTED LOOPS OUTER, preservando publicaciones sin votos.

### Mejoras propuestas

Con mayor volumen de datos el plan actual podría degradarse. Se proponen las siguientes mejoras:

1. **Índice en CONTENIDO(fecha_hora_creacion)**: el filtro `WHERE ct.fecha_hora_creacion >= SYSDATE - 30` aparece como filter en línea 17, lo que significa que Oracle lee todas las filas de CONTENIDO y descarta las que no cumplen. Un índice en esa columna lo convertiría en INDEX RANGE SCAN.

2. **Índice compuesto en CONTENIDO(id_agente, fecha_hora_creacion)**: cubriría tanto el join con AGENTE como el filtro de fecha en una sola estructura, reduciendo accesos a disco.

3. **Con más datos**: el optimizer probablemente cambiaría los nested loops entre tablas grandes a hash joins, que escalan mejor con volúmenes altos.
